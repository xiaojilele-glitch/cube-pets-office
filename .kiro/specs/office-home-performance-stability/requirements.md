# 办公室首页性能稳定性优化需求

## 目标

本规格聚焦桌面办公室首页的刷新与窗口宽度拖拽卡顿问题，只优化前端渲染链路、初始化链路和首屏结构，不改变后端接口、Socket 协议或业务 store 的真相源。

## 需求

### 1. 首屏初始化收口

1. 桌面办公室首页必须由页面壳层统一负责首屏数据准备，避免页面内部多个组件重复触发任务初始化。
2. 驾驶舱组件必须作为展示层使用，不应在挂载时再次发起与首页重复的任务预热。
3. workflow 首屏关键数据与次要上下文数据应拆分加载，次要数据允许延后到首屏稳定后再拉取。

### 2. 视口监听与 resize 稳定性

1. 断点判断必须只在跨越 `mobile / tablet / desktop` 边界时触发订阅更新。
2. 需要连续宽度的页面必须通过单独的宽度 hook 获取数值，并进行 RAF 节流。
3. 系统必须提供 `resize-active` 状态，在用户持续拖拽窗口时只触发开始和结束两类高层状态更新。

### 3. 3D 场景性能档位

1. `Scene3D` 必须支持至少 `balanced` 与 `resizing` 两个性能档位。
2. 在 `balanced` 档位下，桌面场景默认画质应低于现有最高画质，降低 DPR 与阴影成本。
3. 在 `resizing` 档位下，必须临时降低 DPR、关闭 `ContactShadows`，并暂停非关键粒子与装饰型动画。
4. 降级过程中必须保留房间主体、任务焦点、核心交互和必要提示，不得破坏办公室页基本可用性。

### 4. 资产加载与 Html 叠层

1. 场景不应在模块加载时立即预热全部 GLTF 资源。
2. 关键模型可以优先预热，次级家具、宠物与装饰模型必须允许空闲时补齐。
3. 非关键 `Html` 标签、框架标识和装饰型浮签必须支持按需显示或在降级模式中隐藏。
4. 选中 Agent、活跃任务和首屏关键信息必须在降级后仍可见。

### 5. 桌面壳层绘制降级

1. 顶部导航和办公室驾驶舱在 `resize-active` 期间必须切换为低成本视觉外观。
2. 降级样式应减少模糊、强透明和重阴影，但保持原有布局与可操作性。
3. 停止拖拽后，页面应恢复原有玻璃态视觉语言。

### 6. 验收标准

1. 桌面办公室首页硬刷新时，体感应明显比改造前更顺滑。
2. 在 `1280 / 1440 / 1728` 宽度下持续拖拽窗口时，不应出现明显卡死、布局迟滞或长时间掉帧。
3. 任务、Agent、场景三者联动必须保持原有语义，不得出现功能回归。
