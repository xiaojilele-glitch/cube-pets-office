# Launch Operator Surface Convergence - 需求文档

## 概述

当前 office cockpit 首屏上有两块彼此分离的操作区域：

1. `UnifiedLaunchComposer`
   用于任务创建、澄清补问、工作流发起、文件附件以及运行时升级。
2. `TasksCockpitDetail -> OperatorActionBar`
   用于任务执行期的操作，如暂停、恢复、重试、标记阻塞和终止。

从用户视角看，这两块区域本质上都属于“我现在下一步可以做什么”，但它们分别位于底部发起区和右侧详情栏，用户需要在两个区域之间来回切换注意力。

本规格定义一次“界面收敛”改造：将任务操作区并入发起区的提交区域附近，但保持发起路径和任务操作路径在业务层仍然彼此独立。

## 目标

1. 将发起提交与任务操作放入同一个共享操作区。
2. 保持发起动作与任务操作在业务实现上的既有分离。
3. 降低 office cockpit 首屏的扫描成本。
4. 首次实现阶段避免后端接口变更。
5. 保留当前任务操作的规则、确认弹窗和加载行为。

## 非目标

1. 本规格不将 `submitUnifiedLaunch` 与 `submitOperatorAction` 合并为一个后端动作。
2. 本规格不重做整个右侧任务详情面板。
3. 本规格不移除任务详情中的历史、时间线或产物区域。
4. 本规格不修改任务操作的可用性规则。
5. 本规格不改变澄清、工作流发起或运行时升级路由的语义。

## 依赖

- 依赖 `intelligent-launch-convergence`
- 依赖 `mission-operator-actions`
- 依赖 `office-task-cockpit`
- 复用 `task-detail-operations-first` 中的辅助逻辑

## 当前逻辑摘要

### 发起区

`UnifiedLaunchComposer` 当前负责：

- 草稿文本
- 澄清面板渲染
- 附件选择与附件解析
- 发起路由判断（`mission`、`workflow`、`clarify`、`upgrade-required`）
- 提交按钮文案
- 运行时元信息展示

它通过以下入口提交：

- `submitUnifiedLaunch(...)`
- `submitUnifiedClarification(...)`

### 任务操作区

`TasksCockpitDetail` 当前在右侧任务详情栏中渲染一个紧凑版 `OperatorActionBar`。

这块区域负责：

- 主操作判定
- 次级操作和危险操作
- 最近一次操作摘要
- 阻塞上下文
- `mark-blocked` 与 `terminate` 的确认弹窗
- 行内操作反馈与重试反馈

它通过以下入口提交：

- `onSubmitOperatorAction(...)`
- `tasks-store.submitOperatorAction(...)`

### 可行性评估

在不修改后端的前提下，这次收敛是可行的，因为父级页面已经持有全部所需数据：

- 当前激活任务详情
- 当前激活任务 id
- 任务操作 loading 映射
- 发起提交回调
- 任务操作提交回调

因此这次改动属于组件组合与展示层收敛，而不是协议层收敛。

## 用户故事与验收标准

### 1. 底部共享操作区

#### 1.1 作为用户，我希望发起提交和任务操作位于同一块控制区，这样我不需要在两个地方之间来回寻找下一步动作。

- AC 1.1.1: office cockpit 必须在包含发起提交 CTA 的同一底部操作区中渲染任务操作控件。
- AC 1.1.2: 共享操作区必须同时支持“发起模式”交互和“活动任务控制”交互。
- AC 1.1.3: 当没有选中活动任务时，共享操作区仍必须可用。

### 2. 同一界面下保持业务通道分离

#### 2.1 作为系统，即使界面合并，发起动作与任务操作也必须继续沿用当前各自的业务路径。

- AC 2.1.1: 发起提交必须继续使用 `submitUnifiedLaunch(...)`。
- AC 2.1.2: 澄清提交必须继续使用 `submitUnifiedClarification(...)`。
- AC 2.1.3: 任务操作必须继续使用 `onSubmitOperatorAction(...)` 和 `tasks-store.submitOperatorAction(...)`。
- AC 2.1.4: 合并后的 UI 不得要求新增一个统一后的后端接口。

### 3. 任务操作可用性

#### 3.1 作为用户，我希望合并后的操作区继续遵循我现在已有的任务操作规则。

- AC 3.1.1: 任务操作必须基于当前 mission 的 `status` 与 `operatorState` 计算。
- AC 3.1.2: `pause`、`resume`、`retry`、`mark-blocked` 和 `terminate` 必须保留与当前 `OperatorActionBar` 相同的可用性规则。
- AC 3.1.3: 危险操作必须继续与普通操作在视觉上区分开。
- AC 3.1.4: `mark-blocked` 仍必须保留原因填写流程。
- AC 3.1.5: `terminate` 仍必须保留确认流程。

### 4. 提交按钮附近的动作层级

#### 4.1 作为用户，我希望底部右侧动作区能清晰表达哪个是主 CTA，哪些是任务控制动作。

- AC 4.1.1: 提交 CTA 必须继续作为共享操作区中最强的主按钮。
- AC 4.1.2: 任务操作必须紧邻提交 CTA 出现，但作为次级动作组呈现。
- AC 4.1.3: 共享操作区必须通过分组、标签或视觉语气，明确区分发起动作和任务操作。
- AC 4.1.4: 布局必须避免让任务操作看起来像“另一个提交按钮文案”。

### 5. 在底部 dock 中呈现最近任务上下文

#### 5.1 作为用户，我希望在底部 dock 中看到足够的任务操作上下文，这样就不必每次都重新打开右侧卡片。

- AC 5.1.1: 共享操作区必须展示当前激活任务的操作状态。
- AC 5.1.2: 有数据时，共享操作区应该展示最近一次任务操作摘要。
- AC 5.1.3: 合并后的区域中仍必须保留成功与失败的行内反馈。

### 6. 渐进式迁移

#### 6.1 作为产品团队，我们希望在不引入高风险一次性重构的前提下完成控件合并。

- AC 6.1.1: 第一阶段可以先将任务操作移入底部共享区，同时让右侧栏继续承担摘要与历史展示职责。
- AC 6.1.2: 在达到功能对等后，原独立的任务操作卡片应可移除。
- AC 6.1.3: 迁移过程中必须保留当前关于动作判定与操作反馈行为的测试覆盖。

## 约束与风险

1. 底部操作区已经承载澄清与附件相关 UI，必须控制信息密度。
2. `mark-blocked` 和 `terminate` 当前依赖弹窗，迁移到底部区域后不能引入焦点或层级回归。
3. 任务操作只在存在活动 mission 详情时才有效，因此底部发起区在无活动任务时必须优雅降级。
4. 在移动端或窄宽度场景下，任务操作可能需要折叠为溢出菜单或换行模式。
