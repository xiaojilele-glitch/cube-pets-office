# API Fallback Empty States - 需求文档

## 概述

当前前端多个 store 和组件直接调用 `res.json()`，在服务端未连接、返回 HTML fallback 或接口异常时，容易把技术报错直接暴露给用户，例如 `Unexpected token '<'`。同时，多个页面的空态仍然停留在“没有数据”或直接空白。

本 spec 的目标是建立统一的请求兜底、演示模式提示、空态与错误态体系，避免技术细节穿透到用户界面。

## 设计目标

1. 不再向用户裸露 JSON parse 技术报错
2. 服务端不可达时明确提示“演示模式 / 未连接”
3. 空态要解释原因并告诉用户下一步
4. 可恢复错误要提供重试入口

## 依赖关系

- 可独立启动
- 可与 `navigation-convergence`、`task-hub-convergence` 并行

## 非目标

- 本 spec 不改变后端接口协议
- 本 spec 不负责完整视觉统一
- 本 spec 不替代业务级错误提示

## 用户故事与验收标准

### 1. 请求兜底

#### 1.1 作为用户，我希望接口失败时看到可理解的提示，而不是浏览器层技术报错

- AC 1.1.1: 前端公共请求层 SHALL 能识别非 JSON 响应并返回结构化错误
- AC 1.1.2: 用户界面 SHALL 不直接显示 `Unexpected token '<'` 等原始 parse 错误
- AC 1.1.3: 服务端不可达时 SHALL 提示当前处于演示模式或未连接状态

### 2. 空态引导

#### 2.1 作为用户，我希望空白页面告诉我“为什么这里还没有内容”

- AC 2.1.1: 血缘、历史会话、报告、日志等空态 SHALL 说明为空原因
- AC 2.1.2: 空态 SHALL 给出下一步建议或跳转入口
- AC 2.1.3: 空态文案 SHALL 与页面职责一致，不再只写“暂无数据”

### 3. 错误恢复

#### 3.1 作为用户，我希望可恢复错误能直接重试，而不是只能刷新整页

- AC 3.1.1: 可恢复的请求失败 SHALL 提供 retry 入口
- AC 3.1.2: 页面级错误 SHALL 使用统一错误模型或文案策略
- AC 3.1.3: 演示模式与真实错误 SHALL 在文案上有明确区分

## 约束与风险

- 涉及 `workflow-store`、`lineage-store`、`permission-store`、`audit-store` 等多个模块，覆盖面广
- 若高频任务路径文件同时被其他 spec 修改，需要分批接入，避免 worktree 大量冲突
- 统一请求层要足够轻，不要变成又一层难以迁移的大框架
