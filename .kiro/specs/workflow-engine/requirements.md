# 工作流引擎 需求文档

## 概述

工作流引擎是 Cube Pets Office 的核心编排层，负责将用户的自然语言指令通过十阶段管道（方向→规划→执行→评审→元审计→修订→验证→汇总→反馈→进化）自动推进，驱动动态生成的多智能体组织完成协作任务。引擎需要支持服务端和浏览器两种运行时环境。

## 用户故事

### US-1: 用户提交战略指令启动工作流
作为用户，我希望在前端指令面板中输入一条自然语言指令（如"制定本季度用户增长策略"），系统自动创建工作流并开始执行，这样我不需要手动拆解任务。

#### 验收标准
- AC-1.1: 用户通过 `POST /api/workflows` 提交指令文本和可选附件，系统返回工作流 ID
- AC-1.2: 系统为指令生成 `directiveContext`（合并指令文本与附件全文内容）和 `inputSignature`（用于去重）
- AC-1.3: 工作流创建后立即进入 `running` 状态，`started_at` 记录启动时间
- AC-1.4: 管道在后台异步执行，不阻塞 API 响应
- AC-1.5: 如果存在相同 `inputSignature` 且状态为 `running` 的工作流，系统应复用而非重复创建

### US-2: CEO 智能体拆解指令并分发方向
作为系统，我需要让 CEO 智能体分析用户指令，判断需要哪些部门参与，并为每个部门生成具体方向指令，这样任务可以被合理分配到不同团队。

#### 验收标准
- AC-2.1: 系统调用动态组织生成器，根据指令内容创建包含 CEO、经理、Worker 的组织结构
- AC-2.2: CEO 智能体通过 LLM 分析指令，输出结构化的 `CEOAnalysis`（包含各部门方向和参与部门列表）
- AC-2.3: CEO 通过消息总线向每位经理发送对应的方向指令
- AC-2.4: 消息严格遵循层级约束：CEO 只能向经理发送，不能直接向 Worker 发送
- AC-2.5: 方向下发完成后，工作流阶段推进到 `planning`，前端通过 Socket 事件 `stage_change` 收到通知

### US-3: 经理规划任务并分配给 Worker
作为经理智能体，我需要根据 CEO 下发的方向，为下属 Worker 分解并分配具体任务，这样每个 Worker 知道自己要做什么。

#### 验收标准
- AC-3.1: 每位经理调用 LLM 生成 `ManagerPlan`（包含任务描述和 Worker 分配）
- AC-3.2: 多个部门的经理可以并行规划，并发数受 `LLM_MAX_CONCURRENT` 控制
- AC-3.3: 每个任务创建 `TaskRecord`，状态为 `assigned`，关联 `worker_id`、`manager_id`、`department`
- AC-3.4: 经理通过消息总线向各 Worker 发送任务描述
- AC-3.5: 如果经理 LLM 调用失败，记录 `WorkflowIssue` 并尝试使用默认任务分配

### US-4: Worker 并行执行任务产出交付物
作为 Worker 智能体，我需要根据分配的任务描述调用 LLM 产出交付物，这样经理可以评审我的工作成果。

#### 验收标准
- AC-4.1: 同一部门内的 Worker 可以并行执行，跨部门也可并行
- AC-4.2: Worker 调用 LLM 时注入自身 SOUL.md 人设和记忆上下文
- AC-4.3: 交付物写入 `TaskRecord.deliverable` 字段，任务状态更新为 `submitted`
- AC-4.4: Worker 通过消息总线向经理回传交付物
- AC-4.5: 单个 Worker 执行失败时记录 `WorkflowIssue`，不阻塞其他 Worker

### US-5: 经理按 20 分制评审 Worker 交付物
作为经理智能体，我需要按准确性、完整性、可操作性、格式四个维度对 Worker 交付物评分，这样可以量化交付质量并决定是否需要修订。

#### 验收标准
- AC-5.1: 经理调用 LLM 输出 `ReviewScore`（accuracy/completeness/actionability/format 各 0-5 分，total 0-20 分）
- AC-5.2: 评分结果和反馈文本写入 `TaskRecord` 对应字段
- AC-5.3: 评分 ≥16 分标记为通过，10-15 分标记为需修订，<10 分标记为拒绝
- AC-5.4: 评审完成后通过 Socket 事件 `score_assigned` 通知前端
- AC-5.5: 如果 LLM 返回的评分格式无法解析，使用默认评分（每项 3 分，总分 12）并记录 issue

### US-6: 元审计节点独立检查交付物质量
作为系统，我需要让独立的审计节点（非经理）对交付物做合规性和质量检查，这样评审不完全依赖单一经理的判断。

#### 验收标准
- AC-6.1: 系统从动态组织中识别 `audit` 角色的节点
- AC-6.2: 审计节点调用 LLM 检查交付物是否符合 Worker 的 SOUL.md 规范、是否存在 AI 套话
- AC-6.3: 审计反馈写入 `TaskRecord.meta_audit_feedback`
- AC-6.4: 审计结果参与后续修订决策（与经理评审反馈合并）
- AC-6.5: 如果组织中没有审计节点，跳过此阶段并记录日志

### US-7: 低分交付物退回修订并验证
作为系统，我需要将评分低于 16 分的交付物退回给 Worker 修订，并让经理逐条验证反馈是否被回应，这样交付质量可以迭代提升。

#### 验收标准
- AC-7.1: 评分 <16 的任务自动进入修订流程
- AC-7.2: Worker 收到经理反馈和审计反馈，基于此生成 v2 交付物（写入 `deliverable_v2`）
- AC-7.3: 经理逐条确认反馈点是否被回应，输出 `VerifyResult`
- AC-7.4: 如果未回应比例 >30%，可触发 v3 修订（写入 `deliverable_v3`）
- AC-7.5: 修订后任务版本号递增（`version` 字段）

### US-8: 部门汇总与 CEO 总评
作为系统，我需要让经理汇总部门执行结果，CEO 做全局总评，这样用户可以看到完整的执行报告。

#### 验收标准
- AC-8.1: 每位经理调用 LLM 生成部门汇总报告
- AC-8.2: 部门报告包含任务统计、平均评分、关键发现
- AC-8.3: CEO 收到所有部门汇总后生成全局反馈
- AC-8.4: 部门报告以 JSON 和 Markdown 两种格式持久化到经理的 `reports/` 目录
- AC-8.5: 最终综合报告持久化到 CEO 的 `reports/` 目录

### US-9: 工作流完成后触发智能体进化
作为系统，我需要在工作流结束后分析评分数据，自动更新智能体的人设定义和关键词配置，这样智能体可以从每次执行中学习。

#### 验收标准
- AC-9.1: 调用 `EvolutionService.evolveWorkflow()` 分析本轮所有评分
- AC-9.2: 识别弱维度（平均分 <3/5）并生成 SOUL.md 补丁
- AC-9.3: 更新 HEARTBEAT.md 中的关键词权重
- AC-9.4: 通过 `CapabilityRegistry` 注册本轮展示的新能力
- AC-9.5: 进化完成后调用 `materializeWorkflowMemories` 将本轮记忆持久化

### US-10: 工作流全程实时可视化
作为用户，我希望在前端实时看到工作流的执行进度、智能体状态和消息流动，这样我可以理解多智能体协作的过程。

#### 验收标准
- AC-10.1: 每个阶段切换时发射 `stage_change` Socket 事件
- AC-10.2: 智能体开始工作时发射 `agent_active` 事件
- AC-10.3: 智能体间消息发送时发射 `message_sent` 事件（包含预览文本）
- AC-10.4: 评分完成时发射 `score_assigned` 事件
- AC-10.5: 工作流完成/失败时发射 `workflow_complete` 或 `workflow_error` 事件
