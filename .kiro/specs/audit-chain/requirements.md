# 审计链 / 不可篡改日志 需求文档

## 概述

审计链模块负责将系统中的关键决策、权限变更、数据访问、Agent 操作等事件写入不可篡改的 Append-Only Log，确保日志的完整性、真实性和不可否认性。通过密码学哈希链和时间戳验证，实现事件的防篡改和可追溯，满足合规要求（如 SOC2、GDPR、PCI-DSS）并支持事后审计和问题回溯。

## 用户故事

### US-1: 定义审计事件类型和关键操作
作为系统，我需要明确定义哪些操作必须记录到审计链，包括决策、权限变更、数据访问、Agent 执行等，这样确保审计链覆盖所有关键事件。

#### 验收标准
- AC-1.1: 系统定义 `AuditEventType` 枚举，包括 DECISION_MADE、PERMISSION_GRANTED、PERMISSION_REVOKED、DATA_ACCESSED、AGENT_EXECUTED、AGENT_FAILED、CONFIG_CHANGED、USER_LOGIN、USER_LOGOUT、ESCALATION_REQUESTED、ESCALATION_APPROVED
- AC-1.2: 每个事件类型包含 severity（INFO、WARNING、CRITICAL）、category（security、compliance、operational）、description
- AC-1.3: 关键操作（DECISION_MADE、PERMISSION_GRANTED、DATA_ACCESSED）标记为 severity=CRITICAL，必须记录
- AC-1.4: 系统支持扩展事件类型，新增事件类型需要管理员审批
- AC-1.5: 事件类型定义存储到数据库，支持版本控制

### US-2: 采集和记录审计事件
作为系统，我需要在关键操作执行时自动采集事件信息（操作者、操作内容、结果、时间戳、上下文），并写入审计链，这样确保事件的完整性和及时性。

#### 验收标准
- AC-2.1: `AuditEvent` 包含 eventId、eventType、timestamp、actor（userId/agentId）、action、resource、result、context、metadata
- AC-2.2: 系统在以下关键点自动采集事件：决策生成、权限变更、数据查询、Agent 执行、配置变更
- AC-2.3: 事件采集包含完整的上下文信息：sessionId、requestId、sourceIp、userAgent、organizationId
- AC-2.4: 事件采集不阻塞主业务流程（异步写入或缓冲）
- AC-2.5: 采集失败时记录到本地缓冲，定期重试写入审计链

### US-3: Append-Only Log 存储和防篡改
作为系统，我需要将审计事件写入 Append-Only Log，确保日志只能追加不能修改或删除，并通过密码学哈希链防止篡改，这样保证审计日志的不可否认性。

#### 验收标准
- AC-3.1: `AuditLogEntry` 包含 entryId、eventId、previousHash、currentHash、timestamp、signature、nonce
- AC-3.2: 每条日志条目包含前一条的哈希值（形成哈希链），任何篡改都会破坏链的完整性
- AC-3.3: 哈希计算采用 SHA-256，包含事件内容、时间戳、前一条哈希
- AC-3.4: 每条日志条目使用私钥签名（RSA-2048 或 ECDSA），防止伪造
- AC-3.5: 日志存储采用 Append-Only 数据结构（如 LSM Tree、WAL），禁止更新和删除操作
- AC-3.6: 日志存储支持多副本（至少 3 副本），防止单点故障

### US-4: 时间戳验证和可信时间源
作为系统，我需要为每条审计日志添加可信的时间戳，防止时间戳被篡改，这样确保事件的时间顺序和真实性。

#### 验收标准
- AC-4.1: 系统集成可信时间源（如 NTP、TSA - Time Stamping Authority），获取可信时间戳
- AC-4.2: 每条审计日志包含两个时间戳：系统时间戳和可信时间戳
- AC-4.3: 可信时间戳由第三方时间戳服务签名，包含时间戳令牌和签名
- AC-4.4: 时间戳验证时检查系统时间戳和可信时间戳的偏差（允许偏差 < 1 秒）
- AC-4.5: 时间戳异常（如时间倒退）自动告警

### US-5: 审计链完整性验证
作为系统，我需要定期验证审计链的完整性，检测是否存在篡改或丢失的日志条目，这样确保审计链的可信性。

#### 验收标准
- AC-5.1: `verifyAuditChain(startEntryId, endEntryId)` 验证指定范围内的审计链完整性
- AC-5.2: 验证流程：检查哈希链连续性 → 验证每条日志的签名 → 检查时间戳顺序 → 检查日志条目是否丢失
- AC-5.3: 验证失败时返回详细的错误信息（如哪条日志被篡改、哪条日志丢失）
- AC-5.4: 系统定期（如每小时）自动验证审计链，验证结果记录到监控系统
- AC-5.5: 验证失败时自动告警，包含告警级别、受影响的日志范围、建议的恢复措施

### US-6: 审计日志查询和搜索
作为审计人员，我需要查询审计日志，支持按事件类型、操作者、资源、时间范围等条件搜索，这样可以快速定位关键事件。

#### 验收标准
- AC-6.1: `queryAuditLog(filters, pageSize, pageNum)` 支持多条件查询
- AC-6.2: 支持的查询条件：eventType、actor、resource、result、timeRange、severity、category
- AC-6.3: 查询结果包含完整的事件信息和审计链验证状态
- AC-6.4: 查询性能：单次查询 < 500ms（支持百万级日志）
- AC-6.5: 支持全文搜索（如搜索特定的决策 ID 或用户名）
- AC-6.6: 查询操作本身也记录到审计链（谁查询了什么日志）

### US-7: 审计日志导出和报告
作为审计人员，我需要导出审计日志并生成合规报告，这样可以满足外部审计和合规要求。

#### 验收标准
- AC-7.1: `exportAuditLog(filters, format)` 支持 JSON、CSV、PDF 格式导出
- AC-7.2: 导出数据包含完整的事件信息、哈希链、签名、时间戳
- AC-7.3: 导出文件本身包含完整性验证信息（如总哈希、签名），防止导出后被篡改
- AC-7.4: `generateComplianceReport(timeRange, complianceStandard)` 生成合规报告（SOC2、GDPR、PCI-DSS）
- AC-7.5: 合规报告包含审计覆盖范围、关键事件统计、风险事件、合规性评分
- AC-7.6: 报告生成时记录到审计链，包含报告内容的哈希

### US-8: 审计日志保留和归档
作为系统，我需要定义审计日志的保留策略，并支持日志的归档和长期存储，这样满足合规要求的日志保留期限。

#### 验收标准
- AC-8.1: 系统定义 `RetentionPolicy`（包含 retentionDays、archiveAfterDays、deleteAfterDays）
- AC-8.2: 不同事件类型支持不同的保留策略（如 CRITICAL 事件保留 7 年，INFO 事件保留 1 年）
- AC-8.3: 日志归档时生成归档包（包含日志数据、哈希链、签名、时间戳），存储到冷存储
- AC-8.4: 归档包本身包含完整性验证信息，支持离线验证
- AC-8.5: 日志删除前进行最终验证，确保日志已安全归档
- AC-8.6: 删除操作记录到审计链（谁删除了什么日志、删除原因）

### US-9: 审计链与数据血缘的集成
作为系统，我需要将审计链与数据血缘追踪集成，这样可以追踪数据的完整生命周期和所有相关的审计事件。

#### 验收标准
- AC-9.1: 数据血缘节点包含关联的审计事件 ID（如数据访问事件、处理事件）
- AC-9.2: 审计事件包含关联的数据血缘 ID（如访问的数据 ID、处理的数据 ID）
- AC-9.3: `getDataLineageWithAudit(dataId)` 返回数据的完整血缘和所有相关的审计事件
- AC-9.4: `getAuditTrailWithLineage(eventId)` 返回审计事件的完整上下文和相关的数据血缘
- AC-9.5: 支持从审计事件追溯到数据源，从数据源追溯到审计事件

### US-10: 异常检测和告警
作为系统，我需要监控审计日志，检测异常行为（如异常访问、权限滥用、数据泄露迹象），并自动告警。

#### 验收标准
- AC-10.1: `detectAnomalies(timeWindow)` 分析审计日志，检测异常模式
- AC-10.2: 异常检测规则包括：异常访问频率、异常时间访问、权限提升后立即访问敏感资源、多次失败后成功、批量数据导出
- AC-10.3: 异常检测支持机器学习模型（如隔离森林、LSTM），学习正常行为基线
- AC-10.4: 异常事件自动告警，包含异常类型、风险等级、建议的响应措施
- AC-10.5: 告警记录到审计链，包含告警内容和响应状态

### US-11: 审计链与权限模型的集成
作为系统，我需要将审计链与权限模型集成，记录所有权限相关的操作，这样可以追踪权限的变更历史和权限滥用。

#### 验收标准
- AC-11.1: 权限变更操作（授予、撤销、提升、降级）自动记录到审计链
- AC-11.2: 审计事件包含权限变更的详细信息（操作者、被操作者、权限内容、原因、审批人）
- AC-11.3: `getPermissionAuditTrail(agentId, timeRange)` 返回 Agent 的权限变更历史
- AC-11.4: `getPermissionViolationAudit(timeRange)` 返回所有权限违规事件（拒绝的操作、异常权限使用）
- AC-11.5: 支持权限变更的回滚审计（查询"在某个时间点，这个 Agent 的权限是什么"）

### US-12: 审计链前端展示和交互
作为审计人员，我需要在前端看到审计日志的可视化展示，包括事件时间线、事件统计、异常告警，这样可以直观理解系统的审计状态。

#### 验收标准
- AC-12.1: 审计日志面板展示事件时间线（按时间顺序展示关键事件）
- AC-12.2: 支持事件过滤和搜索（按事件类型、操作者、资源、时间范围）
- AC-12.3: 事件详情页展示完整的事件信息、相关的数据血缘、关联的审计链验证状态
- AC-12.4: 支持审计链完整性验证的可视化展示（如哈希链图、签名验证状态）
- AC-12.5: 异常告警面板展示检测到的异常事件、风险等级、建议的响应措施

### US-13: 审计链的分布式验证
作为系统，我需要支持审计链的分布式验证，允许外部审计人员独立验证审计链的完整性，这样增强审计链的可信性。

#### 验收标准
- AC-13.1: 系统提供审计链的公开验证接口（如 REST API），允许外部验证
- AC-13.2: 验证接口返回审计链的完整信息（日志条目、哈希链、签名、时间戳）
- AC-13.3: 提供离线验证工具（如命令行工具、Python 库），支持本地验证
- AC-13.4: 验证工具支持验证指定范围的审计链（如某个时间段的日志）
- AC-13.5: 验证结果包含详细的验证步骤和结果，便于审计人员理解

### US-14: 审计链与合规框架的映射
作为系统，我需要将审计链与合规框架（如 SOC2、GDPR、PCI-DSS）的要求映射，这样可以自动生成合规报告。

#### 验收标准
- AC-14.1: 系统定义 `ComplianceMapping`，将审计事件类型映射到合规要求
- AC-14.2: 支持的合规框架包括 SOC2、GDPR、PCI-DSS、HIPAA、ISO27001
- AC-14.3: 每个合规框架包含关键要求和对应的审计事件类型
- AC-14.4: `generateComplianceReport(framework, timeRange)` 自动生成合规报告，包含覆盖范围、合规性评分、不符合项
- AC-14.5: 合规报告支持导出为标准格式（如 PDF、XLSX），便于提交给审计机构
